# Бот сбора табло вылетов аэропортов Москвы

Собирает в одну таблицу фактически вылетевшие рейсы из Шереметьево (SVO),
Внуково (VKO) и Домодедово (DME). По каждому рейсу: аэропорт, терминал,
время вылета, авиакомпания, направление, гейт. Кодшеринги склеиваются в
одну запись. Запускается на бесплатных GitHub Actions, данные хранятся в
самом репозитории — никаких внешних сервисов настраивать не нужно.

## Что внутри

| Файл / папка | Что делает |
|---|---|
| `src/poll.py` | Опрашивает табло, запускается каждые 10 минут |
| `src/compile.py` | Собирает итог за сутки, запускается в 09:00 MSK |
| `src/parser.py` | Парсер HTML таблицы Яндекс-Расписаний |
| `src/dedupe.py` | Склейка кодшерингов в один борт |
| `src/validate.py` | Аудит «никого не потерять» |
| `data/snapshots/` | Сырые JSONL-снапшоты по тикам |
| `data/daily/` | Итоговые CSV за каждый день + JSON-аудит |
| `.github/workflows/poll.yml` | Расписание для поллера |
| `.github/workflows/compile.yml` | Расписание для дневной сборки |

## Источник данных

Используется `rasp.yandex.ru` — у Яндекса единое табло всех трёх аэропортов
с уже слитыми кодшерингами. У официальных сайтов аэропортов (svo.aero,
vnukovo.ru, dme.ru) контент рисуется JavaScript-ом, для них пришлось бы
запускать headless-браузер; Яндекс отдаёт обычный HTML и парсится
быстро и надёжно.

Нюанс: у уже вылетевших рейсов гейт пропадает (он есть только пока рейс
ещё на табло как ожидаемый). Поэтому бот опрашивает табло раз в 10 минут
и сохраняет снапшоты — днём собирается достаточно наблюдений, чтобы у
каждого рейса был зафиксирован последний гейт до вылета.

## Как поднять это с нуля (подробная инструкция)

Если ты раньше не пользовался GitHub — следуй шагам ровно по порядку.
В каждом шаге написано, что ты должен увидеть.

### Шаг 1. Создать аккаунт GitHub

1. Зайти на <https://github.com>, нажать **Sign up**.
2. Указать email, придумать пароль, выбрать username.
3. Подтвердить email письмом, которое придёт от GitHub.

### Шаг 2. Создать новый репозиторий

1. На главной странице GitHub справа сверху нажми зелёную кнопку **New**
   (или зайди на <https://github.com/new>).
2. **Repository name** — придумай имя, например `moscow-airports-bot`.
3. **Description** — короткое описание, не обязательно.
4. **Public** — обязательно публичный. На приватном GitHub Actions
   ограничены 2000 минутами в месяц, а нам нужно ~4500 запусков в месяц.
   Данные с публичных табло — не секрет, скрывать нечего.
5. **Add a README file** — не ставь галочку (мы зальём свой README).
6. **Add .gitignore** — не ставь.
7. **Choose a license** — на твой выбор (можно None).
8. Нажми **Create repository**.

После этого ты увидишь страницу с инструкцией «Quick setup». Не закрывай её.

### Шаг 3. Залить файлы проекта

Самый простой способ для новичка — через веб-интерфейс GitHub.

1. На странице нового пустого репозитория найди ссылку
   **uploading an existing file** (она появляется в подсказке «…or push an
   existing repository»; если её нет — нажми **Add file → Upload files**).
2. Перетащи туда **все файлы и папки** этого проекта одной кучей:
   - папку `.github/`
   - папку `src/`
   - папку `tests/`
   - папку `data/` (там пустой `.gitkeep`)
   - файл `requirements.txt`
   - файл `.gitignore`
   - файл `README.md`
3. Внизу страницы в поле **Commit changes** введи сообщение, например
   `init`. Нажми **Commit changes**.

> Альтернатива через командную строку (если у тебя установлен git):
> ```bash
> cd moscow-airports-bot
> git init
> git add .
> git commit -m "init"
> git branch -M main
> git remote add origin https://github.com/ТВОЙ_USERNAME/moscow-airports-bot.git
> git push -u origin main
> ```

После загрузки на странице репозитория ты должен увидеть структуру
с папками `.github`, `src`, `tests`, `data` и файлами `README.md`,
`requirements.txt`.

### Шаг 4. Включить GitHub Actions

1. На странице репозитория сверху перейди в раздел **Actions**.
2. В первый раз GitHub спросит «I understand my workflows, go ahead and
   enable them». Нажми эту кнопку.
3. После этого в списке слева ты увидишь два workflow:
   - **Poll airports**
   - **Compile daily CSV**

### Шаг 5. Разрешить Actions писать в репозиторий

Это критично — без этого бот не сможет коммитить собранные данные.

1. На странице репозитория **Settings → Actions → General**.
2. Прокрути вниз до раздела **Workflow permissions**.
3. Выбери **Read and write permissions**.
4. Нажми **Save**.

### Шаг 6. Запустить вручную, чтобы проверить

1. **Actions → Poll airports** (слева в списке).
2. Справа кнопка **Run workflow → Run workflow** (зелёная).
3. Подожди примерно минуту, обнови страницу.
4. Когда задача станет с зелёной галочкой — открой её и посмотри логи.
   Должно быть что-то вроде:
   ```
   [SVO] записал NNN строк в data/snapshots/2026-05-27/0945_SVO.jsonl
   [VKO] записал NNN строк в ...
   [DME] записал NNN строк в ...
   ```
5. Зайди в папку `data/snapshots/` в репозитории — там должны появиться
   файлы.

Если что-то пошло не так — посмотри в самый низ логов, там описание ошибки.
Чаще всего проблема в **Workflow permissions** (Шаг 5).

### Шаг 7. Дальше — ничего

Поллер сам будет запускаться каждые 10 минут. Дневная сборка — в 09:00 MSK.
Никакого вмешательства не нужно.

Свежие CSV каждый день будут появляться в `data/daily/`. Открыть их
можно прямо в браузере (GitHub отрисует таблицу) или скачать.

## Как запустить локально (если хочется проверить парсер на своём компьютере)

Тебе понадобится Python 3.11+ и git.

```bash
# 1. Клонировать репо
git clone https://github.com/ТВОЙ_USERNAME/moscow-airports-bot.git
cd moscow-airports-bot

# 2. Создать виртуальное окружение
python3 -m venv .venv
source .venv/bin/activate   # на Windows: .venv\Scripts\activate

# 3. Поставить зависимости
pip install -r requirements.txt
pip install pytest

# 4. Прогнать тесты парсера
python -m pytest tests/ -v

# 5. Сделать один опрос вручную (создаст файлы в data/snapshots/)
python -m src.poll

# 6. Собрать CSV за конкретную дату
python -m src.compile --date 2026-05-26
```

## Формат итогового CSV

`data/daily/YYYY-MM-DD.csv`, разделитель — запятая, кодировка UTF-8.

| Колонка | Что это |
|---|---|
| `airport` | SVO / VKO / DME |
| `flight_date` | Дата рейса по расписанию (MSK) |
| `scheduled_time` | Время вылета по расписанию (HH:MM) |
| `actual_time` | Фактическое время вылета (если источник его отдал; пока пусто — rasp.yandex.ru даёт только статус) |
| `terminal` | Буква терминала (B/C/D/F/E) |
| `gate` | Номер выхода на посадку (например, 117 или D4) |
| `airlines` | Авиакомпания(и) через `;` |
| `flight_numbers` | Номер(а) рейса через `;` (несколько — это кодшеринг) |
| `destination` | Город прилёта |
| `snapshots_seen` | Сколько раз рейс наблюдался поллером (для отладки) |
| `review` | `1` если требуется ручная проверка (не пойман гейт, странный статус) |
| `review_reason` | Текстовая причина пометки `review` |

Рядом с CSV кладётся `YYYY-MM-DD.audit.json` со сводными метриками:
сколько уникальных рейсов было в снимках, сколько помечено как
вылетевшие, сколько отменили, сколько без гейта — удобно глазами
проверять, не поломалось ли что-то в парсере.

## Известные ограничения и что улучшить позже

- **Фактическое время вылета (`actual_time`) пока всегда пустое.**
  rasp.yandex.ru отдаёт только статус «Вылетел», но не точное время.
  Если нужно — добавим второй источник (radar-24, или официальные сайты
  через Playwright).
- **Гейт может быть пустой**, если рейс задержали мало и поллер не успел
  зафиксировать предстоящий статус. На практике при шаге опроса 10 минут
  такое случается редко; такие рейсы попадают в `review=1`.
- **Зависимость от одного источника.** Если Яндекс-Расписания временно
  лягут — данные за этот промежуток будут потеряны. Можно добавить
  fallback на `radar-24.ru` (TODO).
- **GitHub Actions cron бывает с задержкой 5-15 минут** при высокой
  нагрузке на платформу — это нормально. На наши 10-минутные интервалы
  это влияет слабо.
